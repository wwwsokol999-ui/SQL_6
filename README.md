# Домашнее задание «Резервное копирование баз данных»

## Задание 1. Резервное копирование

### 1.1 Восстановление данных за предыдущий день

Для данного случая подходит ежедневное полное резервное копирование (Full Backup).

Преимущества:

- простое восстановление;
- хранится полная копия базы данных;
- позволяет восстановить состояние БД на конец предыдущего дня.

### 1.2 Восстановление данных за час до сбоя

Для данного случая подходит комбинация полного резервного копирования и журналов транзакций (WAL в PostgreSQL или Binary Log в MySQL).

Преимущества:

- возможность восстановления на определённый момент времени (Point In Time Recovery);
- минимальная потеря данных.

### 1.3 Моментальное переключение на рабочую БД

Да, такой сценарий возможен при использовании репликации.

Например:

- Master-Slave;
- Master-Master;
- кластерных решений.

При отказе основного сервера происходит автоматическое переключение на реплику (Failover), что обеспечивает минимальный простой системы.

## Задание 2. PostgreSQL

### 2.1 Пример резервного копирования и восстановления

Создание резервной копии:

```bash
pg_dump -U postgres mydb > backup.sql
```

Восстановление:

```bash
psql -U postgres mydb < backup.sql
```

### 2.2 Автоматизация процесса

Процесс резервного копирования можно автоматизировать с помощью:

- cron в Linux;
- Планировщика заданий в Windows;
- специализированных систем резервного копирования.

Пример задания cron:

```bash
0 2 * * * pg_dump -U postgres mydb > /backup/mydb.sql
```

Данная команда будет создавать резервную копию ежедневно в 02:00.

## Задание 3. MySQL

### 3.1 Пример инкрементного резервного копирования

Для инкрементного резервного копирования в MySQL используются Binary Log.

Включение бинарных логов:

```ini
[mysqld]
log-bin=mysql-bin
```

Создание полной резервной копии:

```bash
mysqldump -u root -p sakila > full_backup.sql
```

Просмотр бинарных логов:

```bash
mysqlbinlog mysql-bin.000001
```

После восстановления полной резервной копии изменения могут быть применены из бинарных логов.

---

## Вывод

Полное резервное копирование позволяет восстановить базу данных на определённую дату. Использование журналов транзакций и бинарных логов позволяет выполнять восстановление на конкретный момент времени. Репликация повышает отказоустойчивость системы и обеспечивает быстрое переключение на резервный сервер при сбое основного.
